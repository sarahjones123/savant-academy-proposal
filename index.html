Section 01

Who v1 is for

Three distinct user types. Different goals, different relationships with the product, different prioritization logic. Here's how to sequence investment across all three in v1.

📊
Priority 1
Analyst

Builds data pipelines, runs transformations, owns outputs. The power user whose depth of mastery directly determines team ROI and Savant's expansion revenue.

Why first: Deepest engagement surface in the product. When analysts win, they pull in more seats and become internal champions. Build for them first and build deep — this is where learning translates directly into retention.
⚙️
Priority 2
Admin / IT

Connects data sources, manages permissions, owns the deployment. A confused or stuck admin is a blocked team — they unlock everyone else's ability to learn.

Why second: One admin enables 5–20 analysts. Onboarding content here pays compounding dividends. Needs precision over depth — the goal is "connected and stable," not mastery of every setting.
🧾
Priority 3
Accountant

Consumes reports, validates outputs, closes the books. Narrower feature surface — needs confidence and trust in the data, not configuration depth.

Why third: Smaller learning surface but high emotional stakes around accuracy. v1 covers their core consumer workflow well. Advanced content waits for v2, informed by actual usage data.
🏷️

Every course carries a role tag so learners immediately see what's built for them. Tags appear as colored pills in the catalog and power the browse filter — a learner should never have to guess whether content applies to their job.

Section 02

Learning tracks

Five tracks, sequenced by priority. Each module follows the same repeatable format: a 4–5 min concept video → Walnut-powered hands-on demo → short quiz → certification exam at track close.

📋 Consistent module structure reduces learner cognitive load and production overhead. Learners know what to expect; the team knows what to build. The Walnut interactive demos are non-negotiable for procedural workflows — you cannot build confidence in data cleaning by watching alone.

Section 03

What's explicitly out of v1

Building everything at once is not an option. The prioritization reasoning is as important as what's included. Every exclusion below is a deliberate call, not an oversight.

✕
Developer / API track

A developer building on Savant's API is a different buyer motion from an internal finance team. v1 serves adoption, not integration. This content likely serves a tiny v1 user segment — defer to v2 when there's actual signal.

✕
Live webinars and cohort learning

Synchronous formats require scheduling overhead, hosting infrastructure, and facilitator time. Async-first scales to any account size and timezone without coordination. Build the library first; layer live sessions in v2 where the format earns its cost.

✕
Community forum integration

Savant already has a community hub (Circle.so bootcamp). The Academy in v1 links out to it — it does not replicate or replace it. Building a second community surface splits engagement and creates maintenance overhead.

✕
Custom learning paths per account

Account-level customization requires LMS admin tooling, CS coordination, and configuration complexity. Role-based tags accomplish 80% of the personalization value with zero overhead. Custom paths are a v2 enterprise feature.

✕
Localization (non-English content)

Translating video, quizzes, and Walnut demos is expensive and delays launch significantly. English fluency and content quality comes first. Localization decisions should be data-driven — identify which regions actually need it before investing.

✕
In-app learning nudges / product tours

In-product guidance is a product design function, not an Academy function. Conflating them creates unclear ownership. The Academy is the structured learning destination; the product can contextually link to it without the Academy owning in-app UX.

→

All six exclusions above are v2 candidates — but only if usage data and customer signal justify them. The developer track gets reconsidered first if accounts show integration-led adoption. Live sessions get added once drop-off analysis shows where async content isn't sufficient.

Section 04

Measuring success

Four metric categories. The content quality metrics matter to the education team. The product adoption and business signal metrics are what move CS, Sales, and the exec team — bring both into a shared dashboard from day one.

📈
Activation & engagement
  • % of new users who start at least 1 module within 14 days of signup
  • Track completion rate by role (target: >60% of starters finish)
  • Median time-to-first-certification per cohort
  • Module drop-off rate — flag any module above 30% abandonment
🔗
Product adoption impact
  • Feature adoption rate: Academy completers vs. non-completers
  • Time-to-first-meaningful-output by cohort (e.g. first pipeline run)
  • Support ticket volume: Academy users vs. non-Academy users
  • Retention at 30/60/90 days, split by Academy engagement tier
🎓
Certification & content quality
  • Certifications issued per month and month-over-month trend
  • Exam pass rate per track — flag anything below 70%
  • Post-module satisfaction score (1–5 star, per module)
  • Quiz item analysis: questions with >60% wrong-answer rate need revision
💼
Business & NPS signal
  • NPS delta: certified users vs. non-certified at same account tenure
  • Expansion revenue correlation — seats added in high-Academy-engagement accounts
  • % of CS escalations where Academy gap is the identified root cause
  • Champion identification: certified users who refer teammates
🔧

Built on MapleLMS (or equivalent), all four metric categories are reportable natively. The critical instrumentation step: connect LMS completion data to product analytics via user email or ID. This unlocks the adoption-impact metrics — the ones that actually move stakeholders beyond the education team. Set up the shared dashboard before launch, not after.

Section 05

Build sequence & roadmap

A 14-week build to full v1 launch, with a clear boundary between what ships and what waits for data before it gets prioritized.

Weeks 1–4
Foundation
  • Platform setup (MapleLMS)
  • Track 1 — Foundations
  • Track 3 — Admin Essentials
  • Role tagging system
  • LMS ↔ product ID link
Weeks 5–9
Core tracks
  • Track 2 — Analyst Core
  • Track 4 — Accountant
  • Walnut demos for all tracks
  • First certification exam live
  • Metrics dashboard v1
Weeks 10–14
Advanced + iteration
  • Track 5 — Analyst Advanced
  • Drop-off analysis review
  • First content revision cycle
  • Soft launch to pilot accounts
  • Broad rollout
Post-launch
v2 signals
  • Developer track (if signal)
  • Live webinar layer
  • Custom account paths
  • Localization decisions
  • Community integration
